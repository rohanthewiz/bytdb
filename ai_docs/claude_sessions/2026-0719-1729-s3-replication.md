# Session: S3 replication (litestream-style) for bytdb

Session ID: `2340c6d0-c1dc-4854-a3e9-e2e69750fea9`
Date: 2026-07-19 (evening)

## Context / motivation

The church-platform project wants to move to K8s with one site per pod,
replacing Postgres with bytdb. Hot path stays on block storage (PVC with
real fsync); object storage's job is asynchronous replication and
recovery. This session added that: litestream-style WAL shipping to any
S3-compatible store, plus streaming snapshot backup. Doctrine settled up
front: **replica = recovery, not HA** — a restored node comes up from
object-store state with at most one ship-interval of loss; no live
failover.

## Why this was easy here (design insight)

btypedb's log **is** the database — a single append-only file of
CRC-framed records with batch (transaction) atomicity, periodically
rewritten by `Compact()` (snapshot + tail, atomic rename). So:

- Incremental replication = shipping byte ranges of an immutable,
  growing file. No page shadowing, no SQLite-style checkpoint racing.
- A replica assembled by concatenating shipped ranges opens exactly like
  a crash-recovered local file: torn/missing tail chunks cost seconds of
  data, never validity (replay discards incomplete tails/batches).
- Shipping not-yet-fsynced bytes is safe — same superset stance as
  `BackupTo` documents.
- The one invariant-breaker is compaction (file swap → all offsets
  void). Solved with **epochs** (below).
- Post-crash divergence (local truncates a torn tail a follower already
  shipped) is sidestepped by starting a fresh **generation** on every
  process start — an old generation still restores to the valid state it
  captured; the new one supersedes it.

## What was built

### btypedb (v0.6.0, commit `2713edb`)

- `replication.go`: `LogState() (epoch, size, err)` and
  `ReadLogRange(epoch, from, max, w)` + `ErrEpochChanged` sentinel.
  Range reads hold only `compactMu` (like `BackupTo`) — writers never
  blocked; `from > size` fails loudly (follower bookkeeping bug).
- `db.go`: new `fileEpoch uint64`; `compact.go` bumps it at the swap
  point (right after `db.file = tmp`).
- `replication_test.go`: ship-and-restore via tiny (7-byte) chunks
  crossing record boundaries; torn-tail replica opens with pre-tear
  data; epoch change on compact + stale-epoch read refusal; range
  error surface.

### bytdb root module (commit `a815462`)

- `engine.go`: streaming `BackupTo(io.Writer)` (Tier 1 — pipe nightly
  snapshots straight to an object store, no temp file) plus
  `LogState`/`ReadLogRange` passthroughs to the kv layer.
- `engine_replication_test.go`: streamed backup and whole-log ship both
  reopen with catalog + rows intact.

### `replicate` package (new, same module, zero new deps)

- `storage.go`: 4-method `Storage` interface (Put/Get/List/Delete) —
  needs atomic PUT and ascending lexicographic listing.
- `replicator.go`: `Source` interface (satisfied by both `*bytdb.Engine`
  and `*btypedb.DB`); `Replicator` with `Run(ctx)`/`Start()`/`Close()`
  (final flush on shutdown, 30s grace), `ShipNow(ctx)`, `Status()`.
  Options: Interval (5s default = data-loss window), MaxChunkBytes
  (8MB), RetainGenerations (3; clamped ≥2), Prefix (per-site bucket
  sharing), Logf. Generation ID =
  `<utc-timestamp-to-ns>-<4-rand-hex>` (lexicographic = chronological).
  Chunk keys: `<prefix>gen/<id>/<start>-<end>.wlog` with %016x offsets —
  contiguity verifiable from the listing alone. Watermark advances only
  after a successful PUT; the interval loop is the retry policy. Rolls a
  new generation when epoch moves or on first ship; prunes old
  generations at roll time (best-effort).
- `restore.go`: `Restore(ctx, store, prefix, destPath)` — newest
  generation with a contiguous chain from offset 0 wins; chain gaps
  truncate (valid prefix, torn-tail semantics); a chunk whose size
  disagrees with its name fails loudly (never splice shifted bytes);
  temp file + fsync + rename + dir sync; `ErrNoReplica` sentinel.
- Tests: in-memory Storage fake; end-to-end Engine→ship→Restore→
  `bytdb.Open`→row checks; idle-ship idempotence; compaction rollover +
  prune; prefix isolation; empty-generation fallback; Start/Close loop
  final-flush; race-detector clean.

### `replicate/s3` (stdlib-only SigV4 client)

- Path-style default, `VirtualHost` option; ListObjectsV2 with
  continuation-token paging; hand-rolled SigV4 (headers
  host/x-amz-date/x-amz-content-sha256, real payload hash — no
  UNSIGNED-PAYLOAD).
- Key correctness detail: the wire query string is built by the same
  canonical encoder that signs (`canonicalQueryString`), so bytes signed
  == bytes sent (url.Values.Encode would differ on spaces: `+` vs
  `%20`).
- Test fake **re-derives the signature server-side** from the raw
  request, so encoding bugs fail in CI, not against Linode. Also covers
  pagination, key chars needing encoding, config validation, both
  addressing modes.

## Release engineering (the fiddly part)

1. btypedb committed (`2713edb`) and tagged `v0.6.0` locally; bumping
   bytdb's go.mod pins **before the tag was pushed** failed — Go wants
   the published go.mod even in go.work workspace mode. Reverted pins to
   v0.5.0 until user pushed.
2. After user pushed btypedb: proxy.golang.org had **cached the
   negative lookup** from the pre-push attempt → `go mod tidy` 404'd via
   sum.golang.org. Fix: `GOPRIVATE=github.com/rohanthewiz/btypedb go mod
   tidy` (scoped bypass; `GOSUMDB=off` alone broke the toolchain
   download). Recorded go.sum hashes are the real ones.
3. bytdb had **no tags at all**. User chose `v0.6.0` (pairing with
   btypedb) over the recommended v0.1.0. Nested-module ordering: tag
   root `v0.6.0` and push it **first**, then bump `pgwire/go.mod`'s
   bytdb pin from the `v0.0.0` placeholder to v0.6.0 (commit `4a104a0`),
   then tag `pgwire/v0.6.0`. Future releases: same order.

## Church-platform integration sketch (next steps, other repo)

- Open engine → `replicate.New(e, s3store, Options{Prefix:
  "sites/<site>"}).Start()`; `r.Close()` before `e.Close()`.
- Pod cold start: if the PVC has no db file, `replicate.Restore` before
  `bytdb.Open`.
- Nightly full snapshot (optional Tier 1): `e.BackupTo` → bucket PUT;
  existing app S3 machinery works against Linode Object Storage.

## Deliberately not built

Live HA/failover, point-in-time restore to arbitrary chunk (Restore
takes the newest complete generation; stopping the concat early would
give PIT if ever needed), multipart upload (chunks are ≤8MB single
PUTs), client-side retry beyond the tick loop, `#-`/jsonb functions etc.
(unrelated backlog).

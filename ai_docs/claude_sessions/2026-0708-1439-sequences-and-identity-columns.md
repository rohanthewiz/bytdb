## Part 0 — Design decision

Question: auto-incrementing primary keys, sequences, or both?

**Answer: both, staged, over one primitive.** Rationale:
- bytdb speaks the Postgres wire protocol, and in Postgres `SERIAL` *is* a sequence (sugar for `CREATE SEQUENCE` + `DEFAULT nextval(...)`) — so it's not either/or, just which surface ships first.
- ORMs need identity-on-PK + `RETURNING id`; almost nobody hand-creates standalone sequences → identity first, user-visible `CREATE SEQUENCE` deferred (stage 4, if ever).
- The engine already had the pattern: `CreateTable` allocated table IDs from a counter key inside the DDL transaction.
- Single-writer engine makes allocation-inside-the-insert-txn free (Postgres keeps sequences non-transactional only to dodge writer contention). Docs deliberately promise only distinctness + monotonicity, keeping room for ID caching / concurrent writers later.
- **The one real semantic choice**: explicit inserts bump the counter to `max(counter, value+1)` — MySQL semantics, not Postgres — killing the classic duplicate-key-after-restore footgun. `SetSeq` is the setval-style escape hatch.

Agreed staging: (1) counter primitive → (2) identity columns + SERIAL sugar → (3) RETURNING → (4) CREATE SEQUENCE if needed.

## Part 1 — Named sequences (`seq.go`, `seq_test.go`)

- One kv pair per sequence in the system keyspace: `tuple(2, 1, name) -> big-endian uint64` (next value), created lazily at 1. New reserved table ID `sysSeqTableID = 2` (`engine.go`). Sequence names are their own namespace — a table and sequence may share a name.
- Shared helper `nextFromCounter(tx, key, start, what)`: read → corruption check (len != 8 fails loudly; restarting a counter would hand out an ID something already owns) → exhaustion check at `MaxUint64` → write back +1. **`CreateTable`'s table-ID allocation in `ddl.go` was refactored onto it** (same behavior, existing corrupt-seq test still passes).
- API, each as `Engine` one-shot (with reentrancy guard) and `Txn` method: `NextSeq`, `SetSeq` (can move backward — caller's contract; exists for restore/setval), `PeekSeq` (non-allocating, `ok=false` when absent), `DeleteSeq` (recreated sequence restarts at 1).
- Allocation is transactional: rollback returns the value (tested).
- Tests: allocation from 1, per-name independence, reopen durability, rollback, set/peek/delete, corruption via raw kv write, exhaustion, 8 racing goroutines → all values distinct, reentrancy guard.

## Part 2 — Identity columns + SERIAL (`identity.go`, engine + sql edits)

Engine:
- `Column.Identity bool` (int columns only, enforced at `CreateTable`). Counter per identity column at `tuple(2, identitySeqIndexID=2, tableID, colID)` — own index space, can't collide with named sequences, survives renames.
- `insertRow` calls `fillIdentity` before coercion: NULL draws next (error past `MaxInt64` — column is int64); explicit ≥ 0 bumps via new `bumpCounterTo` (never moves a counter backward; absent + bump ≤ 1 is a no-op); explicit negatives never touch the counter.
- Identity implies NOT NULL in both `coerceRow` and `updateRow` (UPDATE to NULL → Postgres-worded not-null violation).
- Cleanup: `DropTable` range-deletes the table's identity counters (they live outside its keyspace — easy to forget); `DropColumn` deletes the dropped column's counter (column IDs never reused). `AddColumn` rejects identity — no backfill story yet.
- Failed insert (e.g. duplicate PK) after the draw burns a value → gap, never a repeat. Comment on `insertRow` updated: "checks before writes" now excepts counter bumps.

SQL layer:
- Parser: `SERIAL`/`BIGSERIAL`/`SMALLSERIAL`/`SERIAL2/4/8` handled in `colDef` *before* `typeName` (they're column-def-only pseudo-types) → int + identity + NOT NULL. `GENERATED BY DEFAULT AS IDENTITY` as a constraint clause; `GENERATED ALWAYS` **rejected** (accepting it would silently drop its no-explicit-inserts promise); `serial NULL` rejected as conflicting (as Postgres).
- `ColDef.Identity` → `bytdb.Column.Identity` in CreateTable/AddColumn mapping (`sql.go`).
- `information_schema.columns` (`syscat.go`): identity → `is_nullable = 'NO'` + `column_default = "nextval('t_col_seq'::regclass)"` — the signal introspecting ORMs key "omit on insert" off.
- Insert paths need no SQL-layer change: column-list inserts already fill unnamed columns as NULL, and the engine allocates on NULL.

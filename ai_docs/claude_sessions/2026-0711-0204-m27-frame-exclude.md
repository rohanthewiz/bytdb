## Scope (Postgres-faithful)

- `EXCLUDE CURRENT ROW | GROUP | TIES | NO OTHERS` on every frame
  mode (ROWS/RANGE/GROUPS). After the bounds pick a row's frame,
  exclusion removes: just the current row (CURRENT ROW), the row and
  its ORDER BY peers (GROUP), or the peers but not the row (TIES).
  NO OTHERS is the explicit default spelling (zero value in the AST,
  so absent clauses need no special casing).
- Exclusion only removes rows the bounds selected — TIES never
  re-admits a current row from outside the frame.
- As in PG, `EXCLUDE` is part of the frame clause: `OVER (ORDER BY k
  EXCLUDE GROUP)` without a frame mode is a syntax error. GROUPS
  *mode* still demands ORDER BY, but EXCLUDE GROUP/TIES do not —
  without ORDER BY the partition is one peer group, so GROUP empties
  every frame (SUM NULL, COUNT 0) and TIES leaves exactly the current
  row.
- Completes the frame matrix; on the window arc only window+GROUP BY
  (and DISTINCT in window calls) remain.

## Implementation

- `sql/ast.go`: `FrameExclusion` (ExcludeNoOthers zero / CurrentRow /
  Group / Ties) with `name()`; `WindowFrame` gained `Exclude`.
- `sql/parser.go`: `windowFrame` parses the four EXCLUDE spellings
  (bad word after EXCLUDE → `p.unexpected("CURRENT ROW, GROUP, TIES,
  or NO OTHERS")`); the old "frame exclusion ... not supported"
  rejection is gone.
- `sql/window.go` (the core):
  - `frameSpec` gained `exclude`; `resolveFrame` copies it through.
  - New `seg{lo, hi}` + `(*framer).segs(n, buf)`: a row's frame is
    now an ordered list of disjoint non-empty segments — the excluded
    run ([n,n+1) for CURRENT ROW, the peer group for GROUP/TIES) is
    clipped out of bounds(n), with TIES keeping a middle sliver
    [max(lo,n), min(hi,n+1)). At most 3 segments; every piece is
    intersected with [lo,hi), which is what makes "never re-admit"
    fall out for free. buf reuse avoids per-row allocs.
  - `segPos(segs, k)` maps the k-th frame row to its partition
    position (NTH_VALUE counts across the hole).
  - `assignValueFrame` walks segments: FIRST = segs[0].lo, LAST =
    last seg's hi-1, NTH via segPos against the summed length; empty
    → NULL as before.
  - `assignAggFrame`: exclusion branches into a recompute path with
    last-segment-list memoization (`slices.Equal`), because the hole
    moves with the current row and defeats the anchored UNBOUNDED
    PRECEDING fast path (the m26 note's "revisit if EXCLUDE lands" —
    resolved by gating exclusion out of both prior paths). Under
    EXCLUDE GROUP whole peer groups share segment lists, so RANGE/
    GROUPS frames still aggregate once per group. A `have bool`
    guards the first row: nil memo vs a genuinely-empty segment list
    must not collide (COUNT of an empty frame is 0, not a stale nil).
  - `windowText` renders ` EXCLUDE ...` after the BETWEEN form when
    not NO OTHERS (EXPLAIN, wire-visible).
- `sql/params.go`, `sql/describe.go`: no changes needed — the m25
  Frame deep-copy is a struct copy, so `Exclude` binds through `$n`
  re-binding for free, and exclusion adds no parameters.

## Gotchas / notes for next time

- Milestone numbering: 27 = frame EXCLUDE. Next arc starts at 28.
- `EXCLUDE` without a frame mode is a *syntax* error (the OVER-clause
  parser only enters windowFrame on rows/range/groups) — that matches
  PG's grammar, don't "fix" it.
- The aggregate memo compares segment lists, not [lo,hi] pairs; under
  EXCLUDE CURRENT ROW/TIES consecutive rows differ (the hole moves),
  so peer-group memo hits only happen for EXCLUDE GROUP — expected,
  PG pays similar costs.
- If frame EXCLUDE ever meets an incremental-aggregation rewrite:
  segs() is the single source of frame membership now; bounds() alone
  is only correct when `fs.exclude == ExcludeNoOthers`.
- Still open on the window arc: window + GROUP BY, DISTINCT in window
  calls.
- Still open elsewhere: standalone `CREATE SEQUENCE`, `pg_attrdef`
  rows (psql `\d` Default column), order-aware index path selection.
